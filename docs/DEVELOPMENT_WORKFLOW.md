> 基于 Harness 最佳实践 + Superpowers 能力设计，版本 1.0

## 一、设计理念

### 1.1 核心原则

| 原则 | 含义 | 实践方式 |
|------|------|----------|
| **先查后改** | 不瞎猜接口或凭记忆推断 | 每次修改前先 Read/Grep/Glob 确认 |
| **知识复用** | 避免重复踩坑、重复决策 | 强制读取 Memory 文件后再开发 |
| **影响先行** | 改动前评估影响范围 | GitNexus impact analysis 必须 |
| **增量验证** | 小步快跑，每步验证 | 每个原子改动后运行测试/验证 |
| **溯源记录** | 记录为什么，而非做了什么 | ADR/PITFALL/PATTERN 沉淀 |

## 三、六阶段详细流程

### 阶段 0：启动 (Session Kickoff)

**目标**：加载项目上下文，明确当前任务优先级

**强制动作**：

```markdown
□ 读取 TODO.md 了解待办事项
□ 读取 DECISIONS.md 了解架构背景
□ 读取 PITFALLS.md 避免已知陷阱
□ 读取 PATTERNS.md 参考最佳实践
□ 检查 GitNexus 索引新鲜度
```

**触发命令**：
```
/read-todo      → 读取 TODO.md 并展示优先级列表
/read-memory    → 读取所有 memory 文件摘要
/check-index    → 检查 GitNexus 索引状态
```

**输出**：
- 当前会话的任务上下文
- 已知约束和陷阱
- 可复用的模式

---

### 阶段 1：理解 (Understand)

**目标**：深入理解代码如何工作，找到改动点

**Superpowers 使用**：

| 场景 | 使用工具 | 示例 |
|------|----------|------|
| "这个组件怎么工作的？" | `gitnexus-exploring` | `/gitnexus How does DetailsPage fetch data?` |
| "谁调用了这个函数？" | `gitnexus_context` | `gitnexus_context({name: "fetchMetrics"})` |
| "某个概念涉及哪些代码？" | `gitnexus_query` | `gitnexus_query({query: "SSE connection"})` |
| "查看某个执行流程" | `gitnexus_process` | `READ gitnexus://repo/PIA/process/ExtractFlow` |

**流程图**：

```
开始理解
    │
    ▼
┌─────────────────────┐
│ 明确理解目标         │
│ "我想了解 X"        │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ GitNexus Query      │
│ 找到相关代码位置     │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ GitNexus Context    │
│ 获取符号完整上下文   │
│ (callers/callees)   │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ Read 实际代码       │
│ 验证理解正确        │
└─────────┬───────────┘
          │
          ▼
    理解完成
```

**禁止行为**：
- ❌ 直接猜测代码行为
- ❌ 凭记忆推断接口签名
- ❌ 只看文件名不读内容

### 阶段 2：设计 (Design)

**目标**：规划实现方案，获得用户确认

**触发条件**：
- 新功能实现（多文件改动）
- 架构调整
- 多种实现方案可选

**流程**：

```
触发设计
    │
    ▼
┌─────────────────────┐
│ EnterPlanMode       │
│ 进入规划模式         │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ 探索代码库           │
│ 收集约束和依赖       │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ 设计多个方案         │
│ 分析 Trade-offs     │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ AskUserQuestion     │
│ 展示方案让用户选择   │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ 写入 Plan 文件      │
│ ExitPlanMode        │
└─────────┬───────────┘
          │
          ▼
    设计确认
```

**设计文档模板**：

```markdown
# 实现计划: [功能名称]

## 背景
[为什么需要这个功能]

## 方案对比

| 方案 | 优点 | 缺点 | 风险 |
|------|------|------|------|
| A: xxx | ... | ... | LOW |
| B: xxx | ... | ... | MEDIUM |

## 推荐方案
方案 A，理由：...

## 实现步骤
1. [ ] 步骤1
2. [ ] 步骤2
3. [ ] 步骤3

## 影响范围
- 文件：xxx.tsx, xxx.ts
- 执行流：XXXFlow
- 风险等级：LOW/MEDIUM/HIGH
```

### 阶段 3：实现 (Implement)

**目标**：安全编写代码，每步验证

**强制动作序列**：

```
┌──────────────────────────────────────────────────────────────┐
│                    实现阶段 Checklist                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  □ 1. Impact Analysis                                        │
│      gitnexus_impact({target: "symbolName",                  │
│                       direction: "upstream"})                 │
│      → 确认影响范围，评估风险                                  │
│                                                              │
│  □ 2. 检查 PATTERNS.md                                       │
│      → 是否有可复用的模式？                                   │
│                                                              │
│  □ 3. 编写代码                                                │
│      → 遵循 CLAUDE.md 约束                                    │
│      → 复用现有接口，不另起炉灶                                │
│                                                              │
│  □ 4. 单元验证                                                │
│      → 相关测试通过                                           │
│      → TypeScript 编译通过                                    │
│                                                              │
│  □ 5. 运行验证 (/verify)                                      │
│      → 启动应用验证效果                                        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**影响分析决策树**：

```
gitnexus_impact 结果
    │
    ├─ d=1 直接调用者 < 5 个
    │       │
    │       └─→ 风险: LOW → 直接实现
    │
    ├─ d=1 直接调用者 5-15 个
    │       │
    │       └─→ 风险: MEDIUM → 谨慎实现，逐个验证
    │
    ├─ d=1 直接调用者 > 15 个
    │       │
    │       └─→ 风险: HIGH → 必须用户确认后才能修改
    │
    └─ 涉及关键路径 (auth, data, extraction)
            │
            └─→ 风险: CRITICAL → 必须用户确认 + 完整测试
```

**代码编写规范**：

```typescript
// ✅ 正确：先确认接口再使用
// 1. Read api.ts 确认接口签名
const response = await api.search(query);

// ❌ 错误：凭记忆猜测
const response = await fetch('/api/search?q=' + query);
```

### 阶段 4：交付 (Deliver)

**目标**：代码审查、简化、提交

**流程**：

```
开始交付
    │
    ▼
┌─────────────────────┐
│ /code-review        │
│ 正确性 + 质量审查    │
└─────────┬───────────┘
          │
          ├─ 发现问题 ──▶ 返回实现阶段修复
          │
          ▼
┌─────────────────────┐
│ /simplify           │
│ 简化重构优化        │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ gitnexus_detect_    │
│ changes()           │
│ 确认改动范围        │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ 更新相关文档        │
│ (CLAUDE.md, etc.)   │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ Git Commit          │
│ 遵循提交规范        │
└─────────┬───────────┘
          │
          ▼
    交付完成
```

**提交规范**：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**类型**：
- `feat`: 新功能
- `fix`: Bug 修复
- `refactor`: 重构
- `docs`: 文档更新
- `test`: 测试相关
- `chore`: 构建/工具链

---

### 阶段 5：沉淀 (Consolidate)

**目标**：将经验转化为可复用知识

**触发条件**：

| 条件 | 写入文件 | 内容 |
|------|----------|------|
| 做出重要技术选型 | DECISIONS.md | ADR 格式记录 |
| 遇到并解决了坑 | PITFALLS.md | 问题+原因+方案+预防 |
| 发现可复用模式 | PATTERNS.md | 场景+代码+优点 |
| 引入新术语 | GLOSSARY.md | 术语+含义+示例 |

**决策树**：

```
经验是否值得记录？
    │
    ├─ 是一次性问题？
    │       └─ 否 → 写入 PITFALLS.md
    │
    ├─ 是重要技术选型？
    │       └─ 是 → 写入 DECISIONS.md
    │
    ├─ 是否可复用？
    │       └─ 是 → 写入 PATTERNS.md
    │
    └─ 是新术语？
            └─ 是 → 写入 GLOSSARY.md
```

## 五、自动化与 Hook

### 5.1 Git Hook 集成

**post-commit** (自动记录提交到 SESSION.md):
```bash
#!/bin/bash
COMMIT_MSG=$(git log -1 --pretty=%B)
COMMIT_HASH=$(git rev-parse --short HEAD)
echo "## Commit: $COMMIT_HASH" >> .claude/memory/SESSION.md
echo "$COMMIT_MSG" >> .claude/memory/SESSION.md
echo "" >> .claude/memory/SESSION.md
```

**pre-push** (推送前检测):
```bash
#!/bin/bash
# 检查是否有需要记录的新知识
# 提示开发者考虑更新 Memory 文件
```

### 5.2 定时任务

| 任务 | 频率 | 说明 |
|------|------|------|
| SESSION.md 归档 | 每周 | 归档 7 天前的会话 |
| Memory 价值评估 | 每月 | 清理过时内容 |
| GitNexus 索引更新 | 每次开发前 | `npx gitnexus analyze` |

## 六、质量门禁

### 6.1 提交前检查

```markdown
## 提交前 Checklist

### 代码质量
- [ ] TypeScript 编译通过
- [ ] ESLint 无错误
- [ ] 相关测试通过

### 约束遵守
- [ ] API 调用已封装到服务层
- [ ] SSE 连接有 cleanup 函数
- [ ] 数据有 evidence 字段
- [ ] 单位已标准化

### 影响确认
- [ ] gitnexus_detect_changes() 已运行
- [ ] 影响范围符合预期
- [ ] 高风险改动已确认

### 文档更新
- [ ] CLAUDE.md 约束已更新 (如有新约束)
- [ ] Memory 文件已更新 (如有新知识)
```

### 6.2 风险门禁

| 风险等级 | 条件 | 动作 |
|----------|------|------|
| LOW | d=1 < 5, 非关键路径 | 直接实现 |
| MEDIUM | d=1 5-15 或 2+ 执行流 | 谨慎实现，逐步验证 |
| HIGH | d=1 > 15 或 3+ 执行流 | 必须用户确认 |
| CRITICAL | 涉及 auth/data/extraction | 用户确认 + 完整测试 |
