## ADR-001: 选择 Tauri 而非 Electron

**日期**：2026-05-03
**状态**：已采纳

### 背景
需要开发桌面应用程序，考虑两种主流方案：
1. **Electron**：成熟生态，但体积大（~150MB）
2. **Tauri**：新兴框架，体积小（~10MB），使用系统 WebView

### 决策
采用 Tauri v2。

### 理由
1. **体积优势**：打包后体积减小 90%+，提升下载体验
2. **性能优势**：使用系统 WebView，内存占用更低
3. **安全性**：Rust 后端，内存安全有保障
4. **技术趋势**：Tauri v2 已稳定，生态逐渐成熟

### 后果
- ✅ 安装包小，用户下载快
- ✅ 运行时内存占用低（~100MB vs Electron ~300MB）
- ⚠️ 社区生态不如 Electron 成熟，部分问题需要自己解决
- ⚠️ Windows 上依赖 WebView2（Win10/11 已预装）

## ADR-002: 选择 SSE 而非 WebSocket 进行进度推送

**日期**：2026-05-03
**状态**：已采纳

### 背景
前端需要实时接收 PDF 解析进度，考虑两种方案：
1. **WebSocket**：双向通信，支持更复杂交互
2. **SSE (Server-Sent Events)**：单向推送，实现简单

### 决策
采用 SSE。

### 理由
1. **需求匹配**：只需要服务端推送进度，不需要客户端主动发消息
2. **实现成本**：SSE 原生支持断线重连，WebSocket 需要自己实现
3. **兼容性**：SSE 在所有现代浏览器中都支持良好
4. **资源消耗**：SSE 基于 HTTP，无需额外协议握手

### 后果
- ✅ 前端代码简洁，使用 `EventSource` API 即可
- ✅ 自动重连机制开箱即用
- ⚠️ 无法实现双向通信（如取消任务），需要额外 HTTP 接口

### 相关代码
- `src/pages/ResultsPage.tsx:89-119`
- `backend/api/extract.py`

## ADR-003: 放弃 Redis/Celery，使用 ThreadPoolExecutor

**日期**：2026-05-03
**状态**：已采纳

### 背景
需要异步处理 PDF 解析任务，常见方案：
1. **Celery + Redis**：功能强大，但需要额外依赖
2. **ThreadPoolExecutor**：Python 内置，轻量级

### 决策
采用 ThreadPoolExecutor。

### 理由
1. **部署简化**：桌面应用打包成 .exe，引入 Redis 会增加 100MB+ 体积
2. **用户友好**：用户无需安装 Redis
3. **性能足够**：单机场景下并发任务数不超过 CPU 核心数
4. **维护成本**：少一个外部依赖，少一个故障点

### 后果
- ✅ 打包体积小，部署简单
- ✅ 无需用户安装额外软件
- ⚠️ 任务状态存储在内存，重启后丢失（已通过 SQLite 持久化解决）
- ⚠️ 无法跨机器分布式处理（本项目不需要）

### 相关代码
- `backend/task_manager.py`

## ADR-004: 选择 SQLite 而非 PostgreSQL

**日期**：2026-05-03
**状态**：已采纳

### 背景
需要本地数据库存储文献元数据和提取结果，考虑：
1. **PostgreSQL**：功能强大，支持并发，但需要安装
2. **SQLite**：单文件数据库，无需安装，适合桌面应用

### 决策
采用 SQLite。

### 理由
1. **零配置**：无需用户安装数据库软件
2. **便携性**：单文件数据库，易于备份和迁移
3. **性能足够**：本项目读多写少，SQLite 性能足够
4. **成熟稳定**：广泛使用，可靠性高

### 后果
- ✅ 用户无感知，数据存储在 `%APPDATA%/PIA/storage.db`
- ✅ 备份简单，复制文件即可
- ⚠️ 并发写入有限制（已通过 WAL 模式优化）
- ⚠️ 不支持跨机器访问（本项目不需要）

## ADR-005: 两阶段 AI Pipeline 设计

**日期**：2026-05-03
**状态**：已采纳

### 背景
需要从 PDF 中提取结构化数据，考虑成本和效率：
1. **单阶段**：直接用高推理模型（如 Claude 3.5 Sonnet）处理全文
2. **两阶段**：先用轻量模型筛选，再用高推理模型深度提取

### 决策
采用两阶段 Pipeline。

### 理由
1. **成本控制**：Stage 1 快速筛选不相关文献，避免浪费昂贵模型
2. **效率提升**：Stage 1 响应快（~2s），用户体验好
3. **灵活性**：两个阶段可以使用不同模型（如 DeepSeek + Claude）

### 设计
- **Stage 1**：轻量级模型（如 DeepSeek-Chat / GPT-4o-mini）
  - 快速阅读摘要
  - 判断是否相关（钙钛矿相关？有性能数据？）
  - 成本：~$0.001/篇

- **Stage 2**：高推理模型（如 Claude 3.5 Sonnet / GPT-4o）
  - 深度提取工艺参数
  - 解析 SI 附件
  - 成本：~$0.01-0.05/篇

### 后果
- ✅ 整体成本降低 60%+
- ✅ 用户体验好，Stage 1 快速返回结果
- ⚠️ 需要维护两个模型配置
- ⚠️ 两阶段切换可能丢失上下文（通过 DOI 缓存解决）

## ADR-006: 前端状态管理：useState vs Redux

**日期**：2026-05-03
**状态**：已采纳

### 背景
前端页面间需要共享状态（如搜索结果、选中的文献），考虑：
1. **Redux/Zustand**：全局状态管理，适合复杂应用
2. **useState + Props**：简单状态提升，适合简单应用

### 决策
采用 `useState` + Props 传递。

### 理由
1. **应用规模**：页面数量少（4 个），状态不复杂
2. **开发效率**：无需引入额外库，学习成本低
3. **可维护性**：状态流向清晰，易于调试

### 后果
- ✅ 代码简洁，无额外依赖
- ⚠️ 状态提升可能导致 props drilling（当前深度为 2，可接受）
- 🔄 如果未来状态变复杂，再考虑迁移到 Zustand

### 未来评估
- [ ] 当 props 传递层级超过 3 层时，重新评估
- [ ] 当需要跨多个组件共享状态时，重新评估

## ADR-007: 本地缓存策略：DOI 级缓存

**日期**：2026-05-03
**状态**：已采纳

### 背景
重复访问同一篇文献时，需要避免重复解析：
1. **DOI 级缓存**：以 DOI 为 key，缓存提取结果
2. **无缓存**：每次重新解析

### 决策
采用 DOI 级缓存。

### 理由
1. **成本节约**：避免重复调用 AI API
2. **响应速度**：缓存命中时秒开
3. **用户友好**：离线也能查看已解析文献

### 实现细节
- 缓存 key：`doi`
- 缓存内容：提取的结构化数据（JSON）
- 存储位置：SQLite `papers` 表
- 过期策略：永不过期（除非用户手动清除）

### 后果
- ✅ 成本显著降低，缓存命中率为 0 Token 消耗
- ✅ 离线可用
- ⚠️ 占用本地存储空间（单篇约 10-50KB，可接受）
- ⚠️ 如果原文献有更新，缓存不会自动失效（科研文献一般不更新）
